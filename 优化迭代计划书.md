# AI Knowledge System Agent 化升级计划书

## 1. 结论

结论：**可以迭代成 Agent 项目，而且当前项目已经具备较好的基础。**

但要明确一点：

- **现在的项目本质上是“RAG 知识库问答系统”**，还不是严格意义上的 Agent 系统。
- 它已经有了 Agent 所需的几个关键底座：
  - 对话入口
  - 知识检索能力
  - 上下文记忆能力
  - 文档解析与向量化能力
  - 管理端与运营数据面板
- **缺的不是 AI 调用能力，而是“决策、工具编排、状态管理、任务执行闭环”。**

所以更准确的升级路径应该是：

> 从“单轮/多轮 RAG 问答系统”升级为“具备工具调用、任务拆解、记忆管理、可观测与可控执行的 Agent 系统”。

## 2. 当前项目现状分析

## 2.1 当前架构

当前系统是一个标准的前后端分离 + AI 微服务架构：

- **前端**：React + Vite
- **后端**：Spring Boot + MyBatis-Plus + MySQL + Redis + Caffeine
- **AI 服务**：FastAPI + LangChain + 向量库（Milvus / FAISS）
- **能力核心**：文档解析、向量检索、问答生成、对话上下文管理、管理后台

从代码结构看：

- Java 后端负责业务编排、用户体系、会话管理、管理端能力
- Python 服务负责解析、检索、LLM 生成
- 前端已经具备用户对话界面和管理端入口

这类架构非常适合继续往 Agent 方向演进，因为：

1. **业务边界已经清晰**
2. **AI 能力已拆分到 Python 服务**
3. **知识库链路已跑通**
4. **会话数据和管理能力已具备**

## 3. 当前距离“Agent 项目”还差什么

虽然当前基础不错，但距离真正的 Agent 系统，还存在几个明显缺口。

## 3.1 缺少“任务规划-执行-观察-再规划”闭环

当前问答流程本质还是：

1. 用户提问
2. 检索文档
3. LLM 生成答案
4. 返回结果

这更像 **RAG Assistant**，而不是 Agent。

Agent 至少应具备：

- 判断任务类型
- 拆解子任务
- 选择工具
- 获取工具结果
- 根据结果继续决策
- 在必要时向用户追问或终止

当前 `python-service/api/routes.py` 里的逻辑仍然是单次请求式流程，不是状态机/图执行流程。

## 3.2 缺少统一 Tool Registry

当前工具能力是“写在代码里的逻辑”，不是“可注册、可描述、可调度的工具系统”。

例如现在已有这些隐式工具：

- 文档解析
- 向量检索
- 生成标题
- OCR 提取

但它们还没有统一的 Tool 抽象，比如：

```text
name: knowledge_search
input_schema: { query, top_k, filters }
output_schema: { chunks, scores, docs }
permission: user/admin
retry_policy: ...
observability_tags: ...
```

没有这层抽象，后续很难做：

- 自动选工具
- 多工具串联
- 审计
- 限权
- 回放与评估

## 3.3 缺少 Agent 状态管理

当前系统有聊天上下文，但还没有完整的 Agent 状态模型，例如：

- 当前任务目标
- 当前计划步骤
- 当前执行状态
- 工具调用轨迹
- 中间结论
- 人工确认节点
- 失败原因与重试信息

也就是说，现在保存的是“聊天记录”，不是“任务执行状态”。

## 3.4 缺少评估与观测体系

如果升级成 Agent，必须能回答这些问题：

- 哪类任务成功率最高/最低？
- 哪个工具最常失败？
- 检索不足还是推理不足？
- 回答正确但执行错误的比例是多少？
- 用户在哪个步骤中断最多？

当前虽然有日志和仪表盘，但还不够支持 Agent 级别的评估闭环。

## 3.5 当前还有一些工程层面的基础问题

这些问题不先收敛，会影响后续 Agent 化升级。

### 3.5.1 存在硬编码路径，影响部署和跨平台

例如：

- `src/main/java/com/demo/aiknowledge/controller/ChatController.java`
- `python-service/core/llm.py`

当前存在：

- Windows 临时图片目录硬编码
- Tesseract 路径硬编码

这会导致：

- 本地可运行，但部署不稳定
- 多环境切换成本高
- Agent 相关工具很难迁移和容器化

### 3.5.2 Python 服务仍偏“脚本式编排”

Python 层当前是可用的，但更偏功能拼接，缺少：

- 统一领域边界
- 明确的工作流编排层
- 工具层与推理层分离
- 可插拔策略层

### 3.5.3 部分容错逻辑会掩盖真实问题

例如 Java 侧文档解析失败时有“本地回退直接标记完成”的逻辑，这对演示友好，但对生产和 Agent 调试不友好。

Agent 系统非常依赖“真实状态”，不能让失败伪装成成功。

## 4. 目标定位：应该升级成什么样的 Agent 项目

建议不要一开始就做“通用大而全 Agent 平台”，而是先做 **面向知识库场景的垂直 Agent 系统**。

推荐目标：

## 4.1 第一阶段目标

先做成一个 **知识库问答 Agent**：

- 能识别问题类型
- 能选择是否检索知识库
- 能在回答前先做问题改写/澄清
- 能调用 OCR / 文档 / 检索 / 摘要工具
- 能输出结构化推理结果和引用结果

## 4.2 第二阶段目标

再做成一个 **知识运营 Agent**：

- 自动分析未命中问题
- 识别知识缺口
- 自动生成补库建议
- 生成知识库质量报告
- 发现重复文档、低质量文档、过期知识

## 4.3 第三阶段目标

最后再扩展成 **多 Agent 协作系统**：

- 问答 Agent
- 检索 Agent
- 知识清洗 Agent
- 运营分析 Agent
- 管理端 Copilot Agent

这样路径更稳，落地性更强。

## 5. 推荐的 Agent 化总体架构

## 5.1 保持现有技术方向

建议保留当前架构主干：

- **Spring Boot** 继续承担业务系统、鉴权、数据管理、管理端 API
- **Python 服务** 承担 Agent Orchestrator、RAG、Tool 调度、工作流编排
- **React 前端** 承担用户对话与管理界面

原因：

1. 你当前 AI 能力已经在 Python 侧
2. LangChain 生态更适合先做 Agent 快速迭代
3. Java 继续做稳定业务底座更合适

## 5.2 新增 Agent 分层

建议把 Python 服务拆成以下层：

### A. Interface Layer

负责：

- `/chat`
- `/agent/run`
- `/agent/stream`
- `/agent/tools/...`
- `/eval/...`

### B. Agent Orchestrator Layer

负责：

- 任务分类
- 计划生成
- 工具选择
- 步骤执行
- 状态更新
- 失败重试
- 终止条件判断

### C. Tool Layer

把现有能力抽成标准工具：

- `knowledge_search`
- `doc_parse`
- `doc_summary`
- `ocr_extract`
- `conversation_memory_read`
- `conversation_memory_write`
- `qa_log_search`
- `knowledge_gap_analyze`

### D. Memory Layer

建议拆成四层：

1. **短期记忆**：本轮对话窗口
2. **会话记忆**：当前会话摘要
3. **用户记忆**：用户偏好、常问主题
4. **知识记忆**：文档、Chunk、摘要、标签

### E. Evaluation & Observability Layer

负责：

- 工具调用日志
- Agent 轨迹日志
- 成功率/时延统计
- 错误归因
- 人工反馈采集

## 6. 优化迭代路线图

## Phase 0：先把当前 RAG 底座做稳

### 目标

先把当前系统从“能跑”提升到“稳定可演进”。

### 重点工作

1. 去掉硬编码路径，全部改为配置化
2. 收敛文档解析状态流转，失败不能直接伪装成功
3. 统一 Python 服务配置、日志、异常格式
4. 明确向量检索分数语义和阈值策略
5. 补齐 API 文档、健康检查和基础监控
6. 把图片/OCR 处理链路做成可配置能力

### 输出物

- 稳定版 RAG 服务
- 明确的错误码和状态码体系
- 可部署的配置模板
- 基础监控面板

### 完成标志

- 文档上传、解析、检索、问答、会话链路稳定
- 本地/测试/生产环境切换不再依赖硬编码
- 日志可定位一次完整请求链路

---

## Phase 1：把现有 AI 能力“工具化”

### 目标

把现在分散在业务代码里的 AI 能力，统一抽象成 Tool。

### 重点工作

1. 建立 Tool Registry
2. 为每个工具定义：
   - 名称
   - 输入 schema
   - 输出 schema
   - 超时
   - 权限
   - 失败策略
3. 把以下能力先改造成工具：
   - 文档解析
   - OCR 提取
   - 向量检索
   - 问题改写
   - 摘要生成
   - 对话记忆读写
4. 引入统一的工具执行日志

### 推荐优先级

优先做最有价值的 5 个工具：

1. `knowledge_search`
2. `question_rewrite`
3. `conversation_memory_read`
4. `conversation_memory_write`
5. `ocr_extract`

### 输出物

- Tool 基础框架
- 第一批标准工具
- 工具调用日志表/日志流

### 完成标志

- Agent 不再直接写死调用某个函数，而是通过 Tool 执行器调用能力

---

## Phase 2：落地单 Agent 工作流

### 目标

先实现一个真正可用的“知识库问答 Agent”。

### Agent 行为建议

该 Agent 需要具备如下执行流程：

1. 判断用户问题类型
2. 判断是否需要追问澄清
3. 判断是否要查知识库
4. 必要时做问题改写
5. 调用检索工具
6. 判断检索结果是否足够
7. 再决定：
   - 直接回答
   - 带引用回答
   - 请求用户补充信息
   - 标记为未命中问题
8. 保存会话记忆和执行轨迹

### 重点工作

1. 引入 Agent State
2. 为一次执行建立 traceId / runId
3. 增加步骤级事件流
4. 支持前端展示：
   - 正在检索
   - 正在分析
   - 正在生成答案
5. 支持人工中断、重试、反馈

### 输出物

- Knowledge QA Agent v1
- Agent 轨迹日志
- 前端 Agent 执行状态可视化

### 完成标志

- 用户看到的不只是“答案”，而是“有步骤感知的智能执行过程”
- 遇到复杂问题时，不再只依赖一次检索 + 一次生成

---

## Phase 3：升级为知识运营 Agent

### 目标

让系统不仅能答问题，还能帮管理员持续优化知识库。

### 可落地的 Agent 能力

#### 1. 未命中问题分析 Agent

分析：

- 哪些问题无答案
- 哪些问题检索召回低
- 哪些问题重复出现
- 哪些问题应该补文档

#### 2. 知识质量巡检 Agent

分析：

- 重复文档
- 低质量切块
- 长期无人访问文档
- 过期文档
- 高引用但低满意度文档

#### 3. 自动报表 Agent

定期生成：

- 热门问题周报
- 未命中问题周报
- 知识库增长周报
- 用户提问趋势分析

### 输出物

- Admin Copilot
- 巡检任务面板
- 自动分析报告

### 完成标志

- 管理端从“人工操作后台”升级成“智能运营后台”

---

## Phase 4：多 Agent 协作

### 目标

从单 Agent 扩展到多 Agent 协作，但只在前几阶段稳定之后再做。

### 建议的多 Agent 角色

1. **Router Agent**：路由任务类型
2. **Retrieval Agent**：负责召回、过滤、重排
3. **Reasoning Agent**：负责归纳和答案生成
4. **Memory Agent**：负责摘要、长期记忆写入
5. **Ops Agent**：负责运营分析与后台建议

### 注意事项

多 Agent 不要为了“炫技”而上。

只有当你出现以下问题时，才值得做多 Agent：

- 单 Agent prompt 过重
- 单 Agent 工具选择不稳定
- 不同任务目标冲突明显
- 管理端和用户端能力差异太大

## 8. 技术选型建议

## 8.1 推荐方案

**推荐继续基于 Python + LangChain 生态演进，但要引入工作流编排思想。**

可选方向：

### 方案 A：当前项目最适合

- Spring Boot 保持不变
- Python 服务继续演进
- 在 Python 中引入更清晰的 Agent 工作流编排
- 先做单 Agent，再逐步扩展

适合你当前项目，因为改造成本最低。

### 方案 B：进一步增强

在 Python 侧引入图式编排框架（如 LangGraph 风格思路）来管理：

- 节点
- 状态
- 条件分支
- 重试
- 人工确认

如果后续任务复杂度持续上升，这会比简单 if/else 更稳。

## 8.2 不建议的方案

暂时不建议一开始就：

- 重写为全新的通用 Agent 平台
- 一口气上多 Agent
- 一口气做自动执行高风险操作
- 一口气做太多工具

因为这样很容易把现在已经成型的 RAG 系统打散。

## 11. 风险与应对

## 风险 1：Agent 化后复杂度陡增

### 应对

- 先单 Agent，后多 Agent
- 先做低风险工具，再做执行型工具
- 所有 Agent 执行必须可追踪、可回放、可中断

## 风险 2：工具调用不稳定导致体验下降

### 应对

- 统一工具超时与重试
- 建立工具失败降级策略
- 前端暴露步骤状态，不让用户“无感等待”

## 风险 3：知识检索质量不足，导致 Agent 决策失真

### 应对

- 优先做检索质量治理
- 引入 query rewrite + rerank
- 增加未命中分析闭环

## 风险 4：日志很多但无法定位问题

### 应对

- 每次 Agent 运行生成唯一 runId
- 每次工具调用生成 toolCallId
- 统一结构化日志与 Trace 视图

---

## 12. 当前最建议优先做的 10 件事

1. 去掉所有硬编码路径，统一配置化
2. 收敛文档解析状态流转，失败不能标记成功
3. 统一 Python 服务异常与返回格式
4. 补齐 Agent 所需的 runId / traceId
5. 把检索、OCR、记忆读写抽成标准工具
6. 增加问题改写与澄清节点
7. 为一次问答记录完整工具轨迹
8. 管理端增加“未命中问题分析”面板
9. 增加知识质量巡检能力
10. 在 Python 侧引入正式工作流编排层

---

## 13. 最终建议

如果你的目标是：

- 做毕业设计 / 项目进阶展示
- 做一个有明显差异化的 AI 系统
- 从 RAG 项目升级为更像“AI 产品”的项目

那么这个项目非常适合继续升级成 Agent 项目。

而且你的最佳路径不是推倒重来，而是：

> 保留现有前后端与 RAG 基础，在 Python AI 服务侧逐步补齐工具层、状态层、工作流层和观测层。

这样做的好处是：

- 风险最低
- 复用率最高
- 演进路径最清晰
- 最容易做出阶段性成果

---

## 14. 一句话判断

**能做，而且值得做。**

但建议的方向不是“把现在的问答接口换个名字叫 Agent”，而是：

**把当前 RAG 系统升级成“有工具、有状态、有记忆、有轨迹、可持续优化”的垂直知识 Agent 系统。**

## 16.3 推荐的 Agent Run 数据流

```text
用户发起问题
   ↓
Spring Boot 创建 runId / conversationId
   ↓
调用 Python Agent Service
   ↓
Orchestrator 初始化 AgentState
   ↓
Planner 判断任务类型与步骤
   ↓
Executor 调用 Tool Registry
   ↓
工具返回结果并写入 ToolCall 轨迹
   ↓
State Manager 更新当前步骤状态
   ↓
LLM 汇总生成最终答案
   ↓
写入 Message / AgentRun / AgentStep / Feedback
   ↓
返回前端并展示执行状态
```

## 16.5 架构落地建议

### 最适合你的做法

- **第一步**：不改动现有聊天主流程，对 Agent 新开一条链路
- **第二步**：让部分流量切到 Agent 问答接口
- **第三步**：验证稳定后，再逐步替换原问答流程

这样可以降低风险，也方便做对比测试。
